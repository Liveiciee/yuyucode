# FAQ

## Can YuyuCode run on a desktop browser?

No. The app depends on `yuyu-server.js` running in Termux for all filesystem and shell operations. A browser-only version has no path to file access or command execution. The architecture was built for Android from the ground up.

## Does it work on iOS?

No. Termux is Android-only. iOS does not permit processes with filesystem and shell access outside of sandboxed apps. There are no plans for an iOS port.

## Can I use my own AI model?

The model list is defined in `src/constants.js`. You can add any OpenAI-compatible endpoint by extending the `MODELS` array and updating `askCerebrasStream()` in `src/api.js`. The function name is historical — it handles any provider that speaks the OpenAI streaming format.

## Why not Claude (Anthropic) as the runtime model?

The API costs money. Cerebras and Groq both offer capable free-tier models with fast enough inference for agent loops. The app was built *with* Claude (every architectural decision and bug fix was worked through at claude.ai), but running it at the cost of a paid API was not a constraint worth accepting.

## Why does the agent stop mid-task?

Most likely causes, in order of frequency:

1. **`yuyu-server.js` was killed** by Android's memory manager. Restart it in Termux. The app checks server health before every loop.
2. **Rate limit hit on both providers.** The UI shows a countdown. Wait it out or switch models manually.
3. **`MAX_ITER` reached.** Default is 6 (normal effort). Use `/effort deep` for complex tasks.
4. **Agent returned no actions and no new data.** The loop exits cleanly when there is nothing left to do. This is expected behaviour.

## Why does `/compact` warn me about accuracy loss?

`/compact` uses a recursive summarisation strategy — it asks a small model to summarise the conversation history into ~300 words. Information is inevitably lost. For long sessions where precise recall matters (e.g. multi-hour refactors), use `/handoff` instead. It generates a structured brief that is saved to `.yuyu/handoff.md` and automatically injected at the start of the next session.

## Why is `vitest@3` pinned and never upgraded?

`vitest@4` crashes with `Illegal instruction` on Snapdragon 680 (ARM64). This is a CPU instruction set issue specific to the test runner's native bindings. The pin is enforced in both `package.json` and `YUYU.md`. Do not attempt to upgrade.

## What is `isolate: false` in `vitest.config.js` and why is it forbidden?

With `isolate: false`, `vi.mock()` and `vi.hoisted()` module mock state leaks between test files — tests can pass in isolation but fail when run together. The config currently uses the default (isolated) mode. Any PR that adds `isolate: false` will break the test suite.

## Can I contribute?

Issues are welcome. Pull requests are not expected — the project is a single-developer personal tool and the bus factor is intentional. The codebase is not structured for onboarding.

## Why are large features concentrated in a few files?

Speed. `useAgentLoop.js` (644 lines) and `FileEditor.jsx` (846 lines) are deliberately kept in single files. `useSlashCommands/` is the exception — it was split into per-domain handler files after growing past 800 lines, which proved necessary for SonarCloud cognitive complexity scores. The split is documented in `YUYU.md` under Architecture Decisions.

## The APK build fails in CI — what happened?

Check these in order:
1. `"overrides": { "rollup": "npm:@rollup/wasm-node" }` must be present in `package.json`
2. All required secrets must be set: `ANDROID_KEYSTORE`, `KEYSTORE_PASSWORD`, `KEY_ALIAS`, `KEY_PASSWORD`
3. The commit message must start with `release:` to trigger the GitHub Release step

If the build passes but no Release appears, the commit message prefix is wrong.

## What is `.yuyu/compressed.md`?

A compressed representation of the entire codebase, generated by `yuyu-map.cjs`. It is injected into the system prompt when the user's message contains keywords like `refactor`, `overhaul`, `codebase`, or `arsitektur`. For most tasks it is not loaded — only the relevant keyword-matched files are injected.

## How does memory work?

YuyuCode maintains three memory pools per project: `user`, `project`, and `local`. After each session, `extractMemories()` parses the conversation and stores facts. Before each agent call, `tfidfRank()` scores memories against the current task using TF-IDF + 14-day age decay, and injects the top 5 into the system prompt. There is no vector database — it is a client-side mini-RAG pipeline.
